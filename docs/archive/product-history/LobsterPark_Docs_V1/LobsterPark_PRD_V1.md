
# 龙虾乐园（Lobster Park）PRD V1.0

- 文档状态：可直接进入方案设计/开发评审
- 版本：V1.0
- 产出时间：2026-03-06
- 适用阶段：企业内部试点版
- 产品定位：OpenClaw 企业级统一管理平台 / 控制台

---

## 1. 项目背景

企业希望基于 OpenClaw 构建一个统一平台，让内部员工无需掌握复杂安装与配置知识，也能够自助创建、配置、运行并维护自己的 OpenClaw 应用实例，从而降低 AI Agent 的使用门槛，提升日常办公效率。

平台暂定名称为：**龙虾乐园**。

龙虾乐园不是 OpenClaw 的替代品，而是 OpenClaw 之上的**控制平面（Control Plane）**，负责：
1. 实例生命周期管理
2. 配置管理与发布
3. 节点管理
4. 监控与审计
5. 权限与租户隔离
6. 企业安全治理

---

## 2. 立项结论

### 2.1 可行性结论

该项目**可行**，并且适合作为“OpenClaw 企业化/平台化”的上层产品来建设。

### 2.2 为什么可行

OpenClaw 本身已经具备以下底层能力，可作为龙虾乐园的引擎基础：
- 统一 Gateway / Control UI / WebChat 能力
- CLI 向导式安装与配置能力
- 严格配置校验机制
- 多 agent、workspace、session 隔离能力
- 多实例运行能力
- 节点（node）连接与调用能力
- 健康检查、doctor、usage tracking 等运维能力

因此，龙虾乐园不需要“重做一套 Agent runtime”，而是应聚焦于：
- 统一控制台
- 统一安全治理
- 统一生命周期管理
- 统一监控审计
- 面向企业用户的可视化配置与交付

### 2.3 关键判断

OpenClaw 原生更偏向“个人/单用户助手 + 本地运行”，而龙虾乐园的目标是“企业内部多用户、多实例、多节点、可治理的平台”。  
因此龙虾乐园的核心价值，不是包装一个页面，而是补足以下企业能力：
- 多租户
- RBAC
- 配置版本与发布
- 企业技能白名单
- 统一监控与审计
- 实例隔离与资源治理
- 平台级升级/回滚/运维

## 3. 产品目标

### 3.1 业务目标

1. 让企业员工可以 5~10 分钟内完成一个 OpenClaw 实例的创建和基础配置
2. 降低安装、配置、升级、排障成本
3. 保证每个用户/实例之间的隔离性
4. 提供可观测、可审计、可治理的平台能力
5. 为未来外部商业化版本预留多租户和标准化产品能力

### 3.2 成功标准（V1）

1. 员工无需命令行即可完成实例创建、配置、启动
2. 平台支持一个租户下多个用户、多个实例、多个节点
3. 管理员可查看实例健康、渠道状态、节点在线情况、基础使用量
4. 配置变更可校验、可发布、可回滚
5. 用户无法访问他人实例的配置、凭证、日志、节点
6. 公共技能市场默认关闭，仅允许企业审核通过的技能来源

> 为了让需求可开发、可验收，必须进行分期。  
> “尽力都实现”会导致边界失控，因此本 PRD 定义 V1 必做范围、V1.5 增强范围、V2 展望范围。

### 4.1 V1 必做范围

1. 企业内部账号登录
2. 租户、用户、角色管理
3. OpenClaw 实例创建/启动/停止/删除
4. 实例隔离运行
5. 模型、channel、agent、skill、基础参数配置
6. 配置校验、保存草稿、发布生效、版本回滚
7. 节点管理（配对申请、审批、在线状态查看）
8. 健康监控、渠道状态、基础使用统计
9. 审计日志
10. 企业技能白名单
11. 平台管理员后台

### 4.2 V1.5 增强范围

1. 实例模板
2. 批量运维（批量重启、批量升级）
3. 更丰富的报表
4. 告警策略中心
5. 实例克隆
6. Agent 模板与预设

### 4.3 V2 展望范围

1. 面向外部客户的商业化多租户 SaaS
2. 计费/套餐/资源额度
3. 公共市场与企业私有市场并存
4. API / Terraform / DevOps 集成
5. 企业私有插件仓库
6. 多环境（开发/测试/生产）治理

---

## 5. 非目标（V1 不做）

1. 不做对外收费、支付、账单
2. 不做开放式公共技能市场
3. 不做复杂的可视化工作流编排器
4. 不做跨实例联动编排
5. 不做渠道协议本身的重实现
6. 不做端侧 node App 的研发（仅管理已存在的 OpenClaw node 能力）
7. 不承诺支持 OpenClaw 所有长尾插件/社区扩展的可视化配置，先支持核心能力和 schema 驱动方式

## 6. 核心产品定义

### 6.1 平台对象模型

#### 6.1.1 租户 Tenant
- 企业/组织级隔离单元
- 一个租户下可有多个用户
- 一个租户下可有多个实例
- V1 内部版默认可先只有一个企业租户，但数据模型必须按多租户设计

#### 6.1.2 用户 User
- 平台登录主体
- 属于一个租户
- 可拥有一个或多个实例
- 可被授予角色

#### 6.1.3 实例 Instance
- 龙虾乐园管理的最小运行单元
- 对应一个独立的 OpenClaw runtime
- 拥有独立配置、独立状态目录、独立工作区、独立密钥、独立日志、独立节点关系
- 一个实例可配置多个 agent、多个 channel account、多个 skills

#### 6.1.4 节点 Node
- 连接到某个实例的 companion device / node
- 有配对状态、在线状态、能力清单、最后在线时间

#### 6.1.5 配置版本 Config Version
- 某次实例配置的快照
- 支持草稿、校验中、校验失败、待发布、发布成功、发布失败、已回滚等状态

#### 6.1.6 技能包 Skill Package
- 企业允许安装/启用的技能
- 来源可以是：
  - 平台内置
  - 企业私有仓库
  - 经审核导入的第三方技能
- V1 中不允许用户直接从公共市场任意安装

## 7. 角色与权限设计

### 7.1 角色定义

#### 7.1.1 平台超级管理员
- 管理所有租户
- 管理平台配置
- 管理实例模板
- 管理技能白名单
- 查看全局监控与审计

#### 7.1.2 租户管理员
- 管理本租户用户
- 管理本租户实例
- 管理本租户节点审批
- 查看本租户监控报表与审计日志

#### 7.1.3 普通员工
- 创建并管理自己有权限的实例
- 编辑实例配置
- 查看自身实例运行情况
- 申请节点配对
- 使用实例

#### 7.1.4 安全/审计角色（可选）
- 只读查看日志、变更、告警、技能审核结果
- 无运行操作权限

### 7.2 RBAC 权限粒度

权限粒度至少拆分为：
- tenant.view / tenant.manage
- user.view / user.manage
- instance.view / instance.create / instance.update / instance.delete / instance.start / instance.stop / instance.restart
- config.view / config.edit / config.publish / config.rollback
- node.view / node.approve / node.reject / node.invoke
- skill.view / skill.enable / skill.disable / skill.review
- audit.view
- monitor.view
- template.manage

## 9. 关键业务规则

### 9.1 隔离规则

1. 每个实例必须独立运行，不得共享以下资源：
   - 配置文件
   - 状态目录
   - workspace
   - 凭证/密钥
   - 日志
   - 会话数据
2. 任意实例间默认不可见、不可读、不可操作
3. 任意用户默认不可访问非授权实例
4. 节点必须绑定到唯一实例；若需迁移，必须先解绑再绑定
5. 技能启用必须受企业白名单约束

### 9.2 安全规则

1. 平台不得直接公开裸露的 OpenClaw Gateway 地址给终端用户
2. 所有敏感配置项（token / secret / key / password）必须脱敏展示
3. 技能安装默认仅允许：
   - 平台内置技能
   - 企业审核通过技能
4. 高风险操作必须记录审计：
   - 创建/删除实例
   - 启停实例
   - 发布配置
   - 回滚版本
   - 节点审批
   - 技能启用/禁用
   - 凭证更新
5. 任何配置发布前必须经过 schema 校验和平台预检查

### 9.3 生命周期规则

1. 实例创建成功后默认进入“未发布配置”或“运行中（默认模板）”状态，二选一
2. 配置修改不直接覆盖运行态，必须形成草稿版本
3. 发布成功后才影响运行态
4. 发布失败不得覆盖当前有效配置
5. 删除实例前必须二次确认，且支持软删除保留期（建议 7 天）

## 10. 核心流程

### 10.1 主流程：创建并使用实例

1. 用户登录平台
2. 进入“我的实例”
3. 点击“创建实例”
4. 选择创建方式：
   - 从空白模板创建
   - 从推荐模板创建
5. 填写基础信息：
   - 实例名称
   - 所属租户（如有权限）
   - 运行规格
   - 隔离级别
6. 平台创建实例 runtime
7. 实例进入“创建中”
8. 创建成功后进入实例详情页
9. 用户完成配置：
   - 模型
   - channel
   - agent
   - skill
   - 安全策略
10. 用户保存草稿
11. 平台执行配置校验
12. 校验通过后允许发布
13. 发布成功后实例进入可用状态
14. 用户通过配置好的 channel 或 Web 调试入口使用实例

### 10.2 节点接入流程

1. 用户在实例详情进入 Nodes
2. 发起节点接入
3. 节点侧完成配对请求
4. 平台收到待审批记录
5. 有权限的用户审批通过/拒绝
6. 通过后节点绑定到实例
7. 平台展示节点在线状态、能力、最近心跳

### 10.3 配置变更流程

1. 用户编辑配置
2. 系统生成新的草稿版本
3. 用户点击“校验”
4. 校验通过 -> 变成“待发布”
5. 用户点击“发布”
6. 发布成功 -> 当前运行配置版本更新
7. 若发布失败 -> 当前生效版本不变
8. 用户可选择回滚到历史版本

## 11. 详细功能需求

## 11.1 登录与身份体系

### 11.1.1 功能说明
平台支持企业员工登录，并根据用户所属租户和角色进行权限控制。

### 11.1.2 功能点
1. 用户登录
2. 退出登录
3. 获取当前用户信息
4. 获取角色与权限
5. 管理员分配用户角色

### 11.1.3 规则
1. V1 建议支持企业 SSO 或统一账号体系
2. 平台不在 V1 自建复杂账号中心
3. 用户进入平台后只能看到授权范围内的数据

### 11.1.4 验收标准
1. 未登录用户不可访问平台页面
2. 不同角色看到的菜单和按钮不同
3. 无权限用户调用受限接口时返回 403

---

## 11.2 租户与用户管理

### 11.2.1 功能说明
支持平台管理员和租户管理员管理组织与成员。

### 11.2.2 功能点
1. 租户列表
2. 租户详情
3. 用户列表
4. 用户邀请/导入（可后补）
5. 角色分配
6. 用户禁用/启用

### 11.2.3 字段
#### 租户
- tenantId
- tenantName
- tenantCode
- status
- createdAt
- updatedAt

#### 用户
- userId
- name
- email / 工号
- tenantId
- role
- status
- createdAt
- lastLoginAt

### 11.2.4 验收标准
1. 租户管理员无法看到其他租户成员
2. 用户状态变更实时影响登录和授权

## 11.3 实例管理

### 11.3.1 功能说明
实例管理是平台核心模块，负责实例的创建、查询、删除、启停、重启、进入详情等操作。

### 11.3.2 实例列表字段
- instanceId
- instanceName
- owner
- tenantName
- runtimeStatus
- healthStatus
- currentVersion
- nodeCount
- createdAt
- updatedAt

### 11.3.3 支持操作
1. 创建实例
2. 查看实例
3. 编辑实例基础信息
4. 启动
5. 停止
6. 重启
7. 删除
8. 复制实例（V1.5）
9. 打开调试入口
10. 查看最近告警

### 11.3.4 创建实例输入项
- 实例名称（必填）
- 所属租户（管理员可选）
- owner（默认当前用户）
- 运行模板
- 资源规格（S/M/L）
- 隔离级别（标准 / 加强）
- 描述（选填）

### 11.3.5 创建实例输出
- instanceId
- 创建任务 ID
- 初始状态 = creating

### 11.3.6 业务规则
1. 实例名在租户内唯一
2. 创建过程中不可重复提交
3. 删除实例前必须校验是否存在未解绑节点
4. 删除实例默认软删除
5. 软删除期内支持恢复（建议 V1.5）

### 11.3.7 验收标准
1. 成功发起创建后可看到“创建中”状态
2. 创建成功后实例具备独立 runtime
3. 失败时可查看失败原因
4. 启停/重启操作有审计记录

---

## 11.4 配置中心

### 11.4.1 功能说明
平台为每个实例提供可视化配置能力，覆盖模型、agent、channel、skills、基础安全项等；同时保留 Raw JSON 应急编辑能力。

### 11.4.2 配置页结构建议
1. 基础设置
2. 模型配置
3. Channel 配置
4. Agent 配置
5. Skills 配置
6. 安全与访问控制
7. 高级配置
8. Raw JSON

### 11.4.3 功能点
1. 加载当前生效配置
2. 编辑草稿
3. 结构化表单编辑
4. Raw JSON 编辑
5. 配置校验
6. 保存草稿
7. 发布
8. 回滚
9. 查看差异（当前 vs 草稿 / 历史版本）
10. 导入模板

### 11.4.4 关键设计原则
1. **优先采用 schema 驱动渲染**，避免平台为每个 channel / plugin 手写表单
2. 对于平台重点支持项（模型、channel、agent、skills）可做“增强表单”
3. 所有敏感字段使用加密存储 + 脱敏显示
4. 无论用结构化表单还是 Raw JSON，最终都要生成规范配置版本

### 11.4.5 配置发布规则
1. 保存不等于生效
2. 发布前必须校验
3. 发布失败不得影响当前运行版本
4. 可查看每次发布结果和错误信息
5. 发布成功后记录版本号

### 11.4.6 验收标准
1. 表单修改可保存为草稿
2. 非法配置无法发布
3. 用户可以回滚到指定历史版本
4. 历史版本保留发布人、发布时间、备注、校验结果

## 11.5 模型配置

### 11.5.1 功能说明
为实例设置 LLM 提供商、默认模型、鉴权信息、故障转移等能力。

### 11.5.2 功能点
1. 选择模型提供商
2. 配置 API Key / OAuth 信息
3. 设置默认模型
4. 设置模型路由/备用模型（增强项）
5. 检查模型凭证状态
6. 查看模型使用概况

### 11.5.3 规则
1. 凭证可来源于：
   - 实例级
   - 租户级共享（需显式授权）
2. 租户级共享凭证不能明文下发给普通员工
3. 凭证更新必须记录审计

### 11.5.4 验收标准
1. 配置后平台可识别凭证有效性
2. 配置异常时给出明确错误信息

## 11.6 Channel 配置

### 11.6.1 功能说明
允许用户为实例配置 OpenClaw 支持的 channel，但平台只负责“配置与状态管理”，不重做协议侧逻辑。

### 11.6.2 功能点
1. 选择 channel 类型
2. 配置 channel account
3. 登录 / 授权状态查看
4. 健康状态查看
5. 开关启用/停用
6. 查看最近错误
7. 测试连通性

### 11.6.3 规则
1. 平台必须支持 conservative 默认值
2. 默认不允许高风险开放策略
3. channel 凭证必须加密存储
4. 若 channel 需要扫码/外部授权，应以跳转、二维码、临时会话等方式处理

### 11.6.4 验收标准
1. 用户能看到每个 channel 的启用状态、登录状态、最近探测结果
2. 连接失败时能查看建议修复信息

---

## 11.7 Agent 管理

### 11.7.1 功能说明
一个实例内允许配置多个 agent，用于不同人格、工作区、会话路由或业务场景。

### 11.7.2 功能点
1. Agent 列表
2. 新建 Agent
3. 编辑 Agent
4. 删除 Agent
5. 绑定 workspace / 路由规则
6. 选择默认 agent
7. 查看 agent 状态

### 11.7.3 字段
- agentId
- name
- description
- workspacePath
- routeBindingSummary
- modelProfile
- status

### 11.7.4 规则
1. agent 是实例内对象，不是平台用户
2. agent 间需逻辑隔离
3. agent 删除前须校验是否被路由规则引用

## 18. 验收标准（整体）

### 18.1 功能验收
1. 一个租户下可创建多个实例
2. 不同用户只能看见自己有权限的实例
3. 实例可以完成创建、配置、发布、启动、停止、重启
4. 节点可以完成申请、审批、查看状态
5. 管理员可查看监控和审计
6. 历史配置版本可回滚
7. 技能启用受白名单控制

### 18.2 安全验收
1. 实例间配置与密钥相互隔离
2. 用户 A 无法访问用户 B 实例
3. 敏感信息不明文展示
4. 所有关键操作可审计
5. 平台不允许直接接入未审核公共技能

### 18.3 运维验收
1. 实例异常可被监控发现
2. 发布失败不影响现网
3. 节点离线可见
4. channel 失败可见
5. 平台能给出基础故障定位信息

## 19. 风险与应对

### 19.1 风险：把 OpenClaw 当成“共享多用户单实例”
- 风险：隔离不足、数据串扰、权限治理复杂
- 应对：V1 以“实例级隔离”为主，不以“单 Gateway 多员工共用”作为默认方案

### 19.2 风险：社区技能供应链安全
- 风险：恶意技能、社工脚本、越权执行
- 应对：默认关闭公共技能市场；建立企业审核白名单

### 19.3 风险：配置复杂度过高
- 风险：用户不会配、配错后无法启动
- 应对：模板化 + schema 驱动 + 严格校验 + 预检查

### 19.4 风险：节点和 channel 的认证链路复杂
- 风险：接入成本高、用户理解困难
- 应对：平台用“引导 + 状态可视化 + 修复建议”降低门槛

### 19.5 风险：上游 OpenClaw 快速演进
- 风险：配置字段变化、兼容性变化
- 应对：优先采用 schema 驱动，减少硬编码表单

## 20. 产品决策总结

### 20.1 最重要的 5 个决策
1. 龙虾乐园是 OpenClaw 的控制平面，不重做 runtime
2. V1 采用“实例级隔离”作为默认架构原则
3. V1 从内部平台做起，但数据模型按多租户设计
4. 配置中心采用“schema 驱动 + 模板增强”的混合模式
5. 公共技能市场默认关闭，只保留企业白名单机制

### 20.2 建议开发顺序

#### 第 1 阶段：底座
- 登录/RBAC
- 实例元数据
- runtime 创建/启停
- 隔离能力
- 基础审计

#### 第 2 阶段：可用
- 配置中心
- 校验/发布/回滚
- 实例详情
- 节点管理

#### 第 3 阶段：可运营
- 监控中心
- 使用统计
- 告警
- 模板中心
- 技能审核

## 21. 给开发团队的落地说明

开发不要把本项目理解为“做一个配置页面”。  
本项目本质是：

**OpenClaw Runtime + 企业控制台 + 运维治理层 + 安全治理层**

V1 只要把下面这条链路打通，就已经有很强价值：

**登录 -> 创建实例 -> 填配置 -> 校验 -> 发布 -> 启动 -> 监控 -> 审计**

这是龙虾乐园 V1 的最小闭环。
